\section{Motivation}
\label{sec:introduction-motivation}

Robotics research and engineering increasingly depend on software infrastructure for simulation, motion planning, dynamics, middleware, and related development workflows. In many projects, these software systems are not peripheral tools; they shape how researchers test algorithms, compare alternatives, reproduce results, and move from a simulated setting toward physical robotic systems. The quality and maintainability of these systems therefore affects both scientific work and practical robotics development.

The robotics software ecosystem is not a single class of products. Some tools are full simulation environments; others are planning libraries, physics engines, dynamics libraries, manipulation frameworks, middleware, or support tools. This variety is useful in practice, but it complicates comparison. A researcher choosing software for a robotics workflow may need to consider installation effort, documentation, repository activity, issue handling, evidence that performance has been considered, and whether the tool fits the intended task. These are software engineering questions as well as robotics questions.

This thesis studies robotics software for motion planning and simulation using a State-of-the-Practice approach. It does not propose a new robotics algorithm, implement a simulator, or benchmark robot performance. Its purpose is to collect and organize evidence about existing software packages in the domain. The evidence comes from public artifacts such as repositories, documentation, publications, project metadata, and related project records.

\section{Research Questions}
\label{sec:research-questions}

The research questions are adapted from the Domain X State-of-the-Practice methodology:

\begin{enumerate}[RQ1.]
    \item What software packages are candidates for a State-of-the-Practice study of robotics software for motion planning and simulation, and how can that candidate set be constructed transparently?
    \item What artifacts, tools, and publicly visible development practices are present in the selected software packages?
    \item What commonalities, variabilities, and parameters of variation characterize the selected robotics software packages as a software family?
    \item What public evidence can be collected for the Domain X measurement categories: installability, correctness and verifiability, surface reliability, surface robustness, surface usability, maintainability, reusability, surface understandability, and visibility/transparency?
    \item If approved developer interview data becomes available, what developer pain points and quality-improvement practices are reported for this domain?
\end{enumerate}

Together, these questions separate the construction of the study set from the later measurement and interpretation work. RQ1 covers candidate discovery and filtering. RQ2 and RQ4 cover the public evidence collected through the measurement template. RQ3 covers the commonality analysis. RQ5 remains conditional until approved interview material is available.

\section{Scope}
\label{sec:introduction-scope}

The working domain for this thesis is robotics software for motion planning and simulation. The project was first discussed in relation to inverse kinematics software, but later project communication established the broader domain used here. The scope includes robotics simulators and closely related software used for planning, dynamics, physics, middleware, and robotic system development.

The scope is also limited by the measurement method. The study depends on evidence from public repositories, documentation, issue trackers, publications, and project metadata. For that reason, open-source availability and repository-based measurability are selection criteria. A tool may be important in robotics practice and still fall outside the measured set if its core components are proprietary, if public repository evidence is unavailable, or if the project is no longer maintained enough to represent current practice.

The selected set is deliberately heterogeneous. Many entries are robotics simulators, but ROS 2, MoveIt!, OMPL, and similar tools have different roles. They remain in scope because motion planning and simulation workflows often depend on middleware, planning libraries, and supporting infrastructure. The comparison therefore treats the selected packages as related members of a domain, not as interchangeable products.

\section{Overview of the Methodology}
\label{sec:methodology-overview}

The methodology follows the State-of-the-Practice approach for a selected domain. Candidate packages were identified from academic survey papers, community-curated software lists, and LLM-generated software lists. The LLM-generated lists were used as an auxiliary discovery source only. They were combined with the other source types and were not treated as authoritative.

Each package appearing in the collected sources was counted once per source to produce a mention count. The count is a source-appearance signal, not a measure of software quality, adoption, research impact, or suitability. After ranking by mention count, the candidate list was filtered using open-source availability, repository-based measurability, and maintenance-related considerations.

The selected packages are then examined in two main ways. The Domain X measurement template organizes repository-based and public evidence into summary information, repository metrics, and nine quality categories. The commonality analysis describes the shared and variable capabilities of the selected packages. If approved interview data becomes available, developer interviews can add a practitioner perspective.

\section{Contributions}
\label{sec:contributions}

This thesis is intended to contribute:

\begin{enumerate}
    \item a scoped State-of-the-Practice study for robotics software for motion planning and simulation;
    \item a documented candidate discovery process based on academic, community-curated, and LLM-assisted sources;
    \item an explanation of how mention count, open-source availability, and repository-based measurability are used to form the study set;
    \item a measurement plan based on the Domain X Appendix B quality categories and public evidence; and
    \item a commonality analysis structure for describing shared capabilities, variations, and parameters of variation across the selected packages.
\end{enumerate}

These contributions are methodological and descriptive. The thesis investigates existing software; it does not assume that one package is best for all robotics tasks.
