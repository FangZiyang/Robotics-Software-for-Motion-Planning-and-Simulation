\section{Overview of Selected Software}
\label{sec:overview-selected-software}

The selected software set covers a robotics software ecosystem rather than one product category. It includes simulation environments, physics and dynamics engines, motion planning libraries, middleware, manipulation tools, and related robotics software. This breadth follows from the project domain: robotics software for motion planning and simulation.

The set comes from the discovery and filtering procedure in Chapter~\ref{ch:methodology}. The mention ranking identified software appearing across the collected sources. Open-source availability and repository-based measurability then determined whether each candidate could be studied consistently. The resulting list includes Gazebo, Webots, Bullet/PyBullet, CoppeliaSim/V-REP, MuJoCo, ODE, ROS 2, OpenRAVE, Drake, DART, CARLA, OMPL, Pinocchio, MoveIt!, and others.

This chapter is not a measurement-results chapter. It describes the selected software set and sets up the commonality analysis. Following the Domain X methodology, the final analysis distinguishes commonalities, variabilities, parameters of variation, and relevant terminology. Detailed package-specific claims are left for points where documentation, repositories, papers, or measurement records support them.

\section{Software Categories}
\label{sec:software-categories}

The selected software can be grouped into broad categories for analysis. These categories are not strict boundaries. Some packages fit more than one role, and some act as infrastructure for other packages. The categories guide the commonality analysis rather than impose a rigid taxonomy.

\subsection{Robotics Simulators}
\label{subsec:robotics-simulators}

Robotics simulators provide environments for representing robots, worlds, sensors, actuators, and interactions before or alongside work with physical systems. In the selected set, this category includes packages primarily known as simulation environments or simulator platforms. The commonality analysis examines how these tools describe robot models, environments, interfaces, supported workflows, and integration with other robotics software.

Examples from the selected set that may fall in this category include Gazebo, Webots, CoppeliaSim/V-REP, AirSim, CARLA, MORSE, Habitat, Flightmare, and related simulator-oriented packages. The final classification should be checked against project documentation before detailed claims are added.

\subsection{Physics Engines and Dynamics Libraries}
\label{subsec:physics-engines-dynamics-libraries}

Physics engines and dynamics libraries support the representation of motion, contact, rigid-body or multibody dynamics, collision, and related physical behavior. Some tools in this category are used directly by robotics researchers; others act as underlying engines for higher-level simulation or planning systems.

Examples from the selected set that may fall in this category include Bullet/PyBullet, MuJoCo, ODE, PhysX open-source SDK, Drake, DART, Simbody, Pinocchio, Project Chrono, Newton Dynamics, and SOFA. The commonality analysis distinguishes packages that provide a user-facing simulation environment from packages that mainly provide lower-level modeling, physics, or dynamics functionality.

\subsection{Motion Planning and Robotics Middleware}
\label{subsec:motion-planning-robotics-middleware}

Motion planning and middleware tools support planning, integration, communication, manipulation, and robotic system development. These packages may not be simulators, but they can be central to workflows that combine planning and simulation. Their inclusion reflects the domain boundary of the project.

Examples from the selected set that may fall in this category include ROS 2, MoveIt!, OMPL, OpenRAVE, OROCOS, MRPT, YARP, OpenRTM-aist, and GraspIt!. These tools do not all provide the same functionality. Some are libraries, some are middleware or frameworks, and some are application-oriented tools. The commonality analysis compares them by role and evidence rather than treating them all as simulators.

\section{Common Functional Capabilities}
\label{sec:common-functional-capabilities}

The commonality analysis identifies capabilities that recur across the selected software set. In the Domain X methodology, these recurring capabilities are commonalities of the software family. Expected themes include representation of robotic systems, support for simulation or planning workflows, APIs or interfaces, installation and usage documentation, and connections to broader robotics ecosystems. These themes still need to be refined using project documentation, repositories, and measurement files.

One likely common concern is model representation. Many robotics tools need a way to represent robots, environments, kinematic structures, dynamic properties, or simulation scenes. Another is interaction with external software, such as middleware, planning stacks, programming language APIs, visualization tools, or data formats. Reproducible use is also likely to matter, especially installation instructions, examples, tutorials, and documented dependencies.

The commonality analysis separates functional capabilities from quality evidence. A package may document an API, but the quality, completeness, and maintainability of that API documentation are measurement questions. Similarly, a package may support a simulation workflow, while the ease of installing or reproducing that workflow belongs in the measurement procedure.

\section{Variability Across Software Packages}
\label{sec:variability-across-software-packages}

The selected packages vary in purpose, scale, interfaces, and intended users. In the Domain X methodology, these differences are variabilities within the software family. Some packages provide graphical simulation environments; others provide libraries or frameworks for developers. Some focus on physics or dynamics, while others focus on planning, manipulation, middleware, or domain-specific simulation. These variations are part of the domain rather than exceptions to it.

Dimensions of variation may include software role, licensing model, supported platforms, programming language interfaces, repository organization, documentation style, installation process, supported robot or environment models, and integration with other robotics tools. For each dimension, the final analysis records parameters of variation, such as possible software roles, supported operating systems, or interface types. Where meaningful, it also records binding time, such as whether a choice is made when selecting, compiling, configuring, or running the software. These dimensions are recorded only when checked evidence supports them.

This variability affects the interpretation of later measurement results. A full simulator and a planning library should not be expected to have identical artifacts, documentation, or user workflows. The discussion chapter should interpret measurements in relation to each package's role in the robotics software ecosystem.

\section{Boundary Cases in the Selected Set}
\label{sec:boundary-cases-selected-set}

Several selected packages are boundary cases because they are not full robotics simulators. ROS 2, MoveIt!, and OMPL are the clearest examples identified in the project materials. They are included because the thesis scope is robotics software for motion planning and simulation, not only robotics simulators.

These boundary cases clarify the domain definition. A robotics workflow for motion planning and simulation often depends on middleware, planning libraries, manipulation frameworks, and simulator components working together. Excluding every non-simulator tool would make the study simpler but less representative of the software ecosystem described in the project materials.

They also require careful comparison. Middleware, planning libraries, and simulators do not provide the same user-facing functionality. The commonality analysis compares them through evidence appropriate to their roles and identifies both shared concerns and role-specific variation.
