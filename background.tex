\section{Robotics Software for Motion Planning and Simulation}
\label{sec:robotics-software-background}

Robotics software for motion planning and simulation supports the design, testing, evaluation, and deployment of robotic systems. Motion planning software addresses the problem of generating feasible robot motions under constraints such as geometry, kinematics, dynamics, collisions, and task requirements. Simulation software provides virtual environments in which robots, sensors, actuators, controllers, and physical interactions can be represented before or alongside work with physical hardware.

The domain considered in this report is broader than a single class of software. It includes full simulation environments, physics and dynamics engines, motion planning libraries, manipulation frameworks, middleware, and related development tools. These systems may be used together in a robotics workflow even when they are not direct substitutes for one another. For example, a simulator may use a physics engine, a planning stack may use a collision checking library, and middleware may connect planning, control, perception, and simulation components.

This breadth creates a challenge for a State-of-the-Practice study. A narrow definition of the domain would make comparison easier, but it could exclude important software that practitioners use in closely related workflows. A broad definition captures the practical ecosystem more accurately, but it requires careful discussion of scope. This report therefore treats robotics software for motion planning and simulation as a domain with several related software categories rather than as a single product type.

\section{State-of-the-Practice Studies}
\label{sec:state-of-practice-background}

A State-of-the-Practice study examines how a domain is currently represented in existing tools, artifacts, processes, and evidence. In this project, the domain is robotics software for motion planning and simulation. The study follows a methodology developed for assessing the state of practice in a selected domain, referred to in the project materials as the Domain X methodology.

The State-of-the-Practice perspective differs from a conventional implementation thesis. Rather than designing a new simulator or planner, the study asks what can be learned from existing software systems. This makes the quality of evidence especially important. Claims about software activity, documentation, issue management, installation behavior, or maintainability must be tied to observable project artifacts. Where evidence is missing or ambiguous, the report should state the limitation rather than fill the gap with assumptions.

The Domain X methodology is more specific than a general report outline. It calls for identifying a suitable domain, involving a domain expert, constructing and filtering a candidate software list, performing a domain analysis, gathering source code and documentation, collecting repository metrics, applying a measurement template, surveying developers when approved data collection is available, using the Analytic Hierarchy Process (AHP) to support ranking, and then answering the research questions. This report follows that methodological structure while adapting the candidate discovery step to the project materials.

Prior State-of-the-Practice work in related scientific software domains provides useful structure for this report. The medical imaging and lattice Boltzmann examples organize their reports around domain scope, candidate selection, software quality measurement, research questions, recommendations, and threats to validity. This report follows that general pattern, but the primary measurement vocabulary should come from the Domain X methodology and its Appendix B measurement template.

\section{Open Source Software and Repository-Based Measurement}
\label{sec:open-source-repository-measurement}

Open-source availability is central to this study because the measurement process depends on public evidence. Public repositories can provide evidence about commit history, release activity, issue tracking, documentation, source code organization, testing practices, dependencies, and community interaction. Public documentation and project websites can provide additional evidence about installation, usage, supported platforms, and stated design goals.

Repository-based measurement also has limitations. Repository metadata is a snapshot of a project at the time it is collected. For example, the last commit date changes as a project evolves, and issue counts may change as maintainers open, close, or migrate issues. Some projects use multiple repositories or external issue trackers, which can complicate measurement. Some projects may have active development that is not fully visible in a single public repository. These limitations mean that repository-based measurements must be recorded consistently and interpreted cautiously.

The dependence on public evidence also affects software selection. Proprietary tools or tools with closed core components may be important in practice but difficult to measure using the same procedure as open-source tools. The project materials identify MATLAB/Simulink, Unity, NVIDIA Isaac Sim, and RaiSim as examples of important tools whose inclusion would be problematic for a repository-mining study because their core software is not open source in the required sense. The report therefore distinguishes domain importance from measurement feasibility.

\section{Software Quality Attributes}
\label{sec:software-quality-attributes}

The State-of-the-Practice methodology considers software quality from the perspective of observable evidence. For this project, the primary quality vocabulary should follow the Domain X Appendix B measurement template. The template contains 108 questions grouped under summary information, repository metrics, and nine quality categories: installability; correctness and verifiability; surface reliability; surface robustness; surface usability; maintainability; reusability; surface understandability; and visibility/transparency.

The word "surface" is important in the methodology. Surface reliability, surface robustness, surface usability, and surface understandability are not full experimental evaluations of those qualities. They are evidence-based checks that can be performed with limited time and public artifacts. For example, surface usability may be informed by the presence of a getting-started tutorial, a user manual, documented user characteristics, and support channels. Surface understandability may be informed by a review of selected source files and visible coding conventions. These checks provide structured evidence, but they should not be presented as complete user studies or full quality audits.

Some measurements sit outside the nine top-level quality categories. Repository metrics record facts such as stars, forks, watchers, pull requests, developers, issues, release dates, last commit date, programming languages, lines of code, comments, blank lines, binary files, added and deleted lines, and commit activity. Performance evidence is also part of the template, but it appears as a summary-information question about whether speed, storage, or throughput appears to have been considered. It should therefore not be treated as a separate top-level quality category unless the thesis clearly marks it as an additional observation.

The quality attributes in this report should not be interpreted as complete evaluations of each software package. They are measurement categories for organizing public evidence. A package may score differently under a different use case, platform, user background, or measurement date. Later chapters should therefore report both measurements and the limits of those measurements.

\section{Commonality Analysis}
\label{sec:commonality-analysis-background}

Commonality analysis is used to identify features, assumptions, and requirements that are shared across members of a software family, as well as the dimensions along which those members vary. In the Domain X methodology, the domain analysis is expected to produce commonalities, variabilities, parameters of variation, and common terminology. Parameters of variation should also record when a variation is bound when that timing is meaningful, such as at specification time, build time, or run time. In this report, the commonality analysis helps describe robotics software for motion planning and simulation as a domain. It complements repository-based measurement by focusing on what the selected tools are intended to do and how their roles differ.

For this project, the commonality analysis should be evidence-based. Common capabilities and variations should be derived from documentation, repositories, papers, source materials, or other checked evidence. The analysis should avoid assigning unsupported features to software packages. Where a capability appears likely but has not yet been verified, the report should mark it as needing confirmation.

The commonality analysis is especially useful because the selected software set is heterogeneous. Some tools are full simulators; others are physics engines, planning libraries, middleware, or related robotics software. A commonality analysis allows the report to explain that heterogeneity directly. Rather than forcing every tool into the same category, the report can identify shared robotics software concerns and then describe meaningful variations across categories.
