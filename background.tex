\section{Robotics Software for Motion Planning and Simulation}
\label{sec:robotics-software-background}

Robotics software for motion planning and simulation supports the design, testing, evaluation, and deployment of robotic systems. Motion planning software generates feasible robot motions under constraints such as geometry, kinematics, dynamics, collisions, and task requirements. Simulation software provides virtual environments for representing robots, sensors, actuators, controllers, and physical interactions before, or alongside, work with physical hardware.

The domain in this thesis is broader than a single class of software. It includes full simulation environments, physics and dynamics engines, motion planning libraries, manipulation frameworks, middleware, and related development tools. These systems often appear together in a robotics workflow even when they are not substitutes for one another. A simulator may use a physics engine, a planning stack may use a collision checking library, and middleware may connect planning, control, perception, and simulation components.

This breadth creates a scope problem for a State-of-the-Practice study. A narrow definition would make comparison easier, but it would leave out software that practitioners use in the same workflows. A broader definition better matches the practical ecosystem, provided the thesis is clear about software roles. For that reason, robotics software for motion planning and simulation is treated here as a domain with several related categories rather than as one product type.

\section{State-of-the-Practice Studies}
\label{sec:state-of-practice-background}

A State-of-the-Practice study examines existing tools, artifacts, processes, and evidence in a domain. Here, the domain is robotics software for motion planning and simulation. The study follows the project materials' Domain X methodology for assessing software practice in one selected domain.

This perspective differs from a conventional implementation thesis. The goal is not to design a simulator or planner, but to learn from existing software systems. As a result, claims about activity, documentation, issue management, installation behavior, or maintainability must come from observable project artifacts. When evidence is missing or ambiguous, the thesis records the limitation instead of filling the gap with assumptions.

The Domain X methodology gives more than a chapter outline. It calls for identifying a suitable domain, involving a domain expert, constructing and filtering a candidate software list, performing domain analysis, gathering source code and documentation, collecting repository metrics, applying a measurement template, surveying developers when approved data collection is available, using the Analytic Hierarchy Process (AHP) to support ranking, and then answering the research questions. This thesis follows that structure while adapting candidate discovery to the available project materials.

Prior State-of-the-Practice reports in medical imaging and lattice Boltzmann software provide useful examples of organization. They use chapters on domain scope, candidate selection, software quality measurement, research questions, recommendations, and threats to validity. This thesis follows that general pattern, while taking its measurement vocabulary from the Domain X methodology and its Appendix B template.

\section{Open Source Software and Repository-Based Measurement}
\label{sec:open-source-repository-measurement}

Open-source availability matters because the measurement process depends on public evidence. Public repositories can show commit history, release activity, issue tracking, documentation, source code organization, testing practices, dependencies, and community interaction. Public documentation and project websites can add evidence about installation, usage, supported platforms, and design goals.

Repository-based measurement also has limits. Repository metadata is a snapshot. Last commit dates, issue counts, release information, and pull requests can change after measurement. Some projects use multiple repositories or external issue trackers, and some development activity may not be visible in a single public repository. For these reasons, the measurement record needs clear dates, repository choices, and interpretation rules.

The dependence on public evidence also affects software selection. Proprietary tools, or tools with closed core components, may be important in practice but difficult to measure with the same procedure as open-source tools. The project materials identify MATLAB/Simulink, Unity, NVIDIA Isaac Sim, and RaiSim as examples. Their exclusion is about measurement feasibility, not about their value to robotics.

\section{Software Quality Attributes}
\label{sec:software-quality-attributes}

The State-of-the-Practice methodology treats software quality as something measured through observable evidence. This thesis uses the Domain X Appendix B measurement template as its main quality vocabulary. The template contains 108 questions grouped under summary information, repository metrics, and nine quality categories: installability; correctness and verifiability; surface reliability; surface robustness; surface usability; maintainability; reusability; surface understandability; and visibility/transparency.

The word "surface" is deliberate. Surface reliability, surface robustness, surface usability, and surface understandability are not full experimental evaluations. They are limited checks that can be performed from public artifacts within a feasible amount of time. Surface usability, for example, may be informed by the presence of a getting-started tutorial, a user manual, documented user characteristics, and support channels. Surface understandability may be informed by selected source files and visible coding conventions.

Some measurements sit outside the nine top-level quality categories. Repository metrics record facts such as stars, forks, watchers, pull requests, developers, issues, release dates, last commit date, programming languages, lines of code, comments, blank lines, binary files, added and deleted lines, and commit activity. Performance evidence is also part of the template, but it appears as a summary-information question about whether speed, storage, or throughput has been considered. It is not one of the nine quality categories.

The quality attributes organize public evidence; they are not complete evaluations of each package. Scores may change with the use case, platform, user background, or measurement date. Later chapters therefore need to report both the measurements and their limits.

\section{Commonality Analysis}
\label{sec:commonality-analysis-background}

Commonality analysis identifies features, assumptions, and requirements shared by members of a software family, along with the dimensions on which members differ. In the Domain X methodology, the domain analysis produces commonalities, variabilities, parameters of variation, and common terminology. When timing matters, a parameter of variation can also record binding time, such as specification time, build time, or run time. In this thesis, commonality analysis helps describe robotics software for motion planning and simulation as a domain. It complements repository-based measurement by focusing on what the selected tools do and how their roles differ.

The commonality analysis must remain evidence-based. Common capabilities and variations should come from documentation, repositories, papers, source materials, or other checked evidence. If a capability appears likely but has not been verified, it should remain marked as needing confirmation.

This matters because the selected set is heterogeneous. Some tools are simulators; others are physics engines, planning libraries, middleware, or related robotics software. A commonality analysis makes that heterogeneity explicit. It identifies shared robotics software concerns without forcing every package into the same category.
