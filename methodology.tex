\section{Domain Selection}
\label{sec:domain-selection}

The selected domain for this report is robotics software for motion planning and simulation. This domain was chosen to support a State-of-the-Practice study of software systems used in robotics workflows, including simulation, planning, dynamics, middleware, and related support tools. The project materials indicate that the scope was initially discussed in relation to inverse kinematics software, but later communication established the broader working domain used in this report.

The domain boundary is intentionally broader than "robotics simulators" alone. Many important robotics workflows combine simulation environments with planning libraries, physics engines, middleware, and supporting tools. Restricting the study to full simulators would simplify comparison, but it would also omit software that is central to motion planning and simulation practice. The broader scope therefore better matches the project goal, provided that the report clearly distinguishes different software roles.

Domain selection was also constrained by the Domain X State-of-the-Practice methodology. The methodology expects a domain with a stable technical basis, an identifiable community, open-source software options, and enough candidate packages to support comparison. The robotics software domain satisfies these requirements at the level needed for this thesis because motion planning and simulation are established robotics topics and because the project materials identify a substantial set of candidate packages with public artifacts.

The methodology also expects a domain expert to help vet the software list, domain analysis, and later ranking. The project materials identify Dr. Matthew Giamou as the domain expert involved in scope and candidate-list discussions. Where further expert vetting is still needed, this thesis should mark the relevant material as pending rather than presenting it as final.

\section{Candidate Software Discovery}
\label{sec:candidate-software-discovery}

Candidate software discovery used three source categories: academic sources, community-curated sources, and LLM-assisted sources. The purpose of using multiple source categories was to produce a broad initial candidate list without relying on a single discovery method. The resulting list was then consolidated and filtered before measurement.

\subsection{Academic Sources}
\label{subsec:academic-sources}

Academic sources were used to identify software packages that appear in research discussions of robotics simulators, simulation tools, and related challenges. The source catalogue includes survey-style or comparative papers that mention packages such as Gazebo, Webots, V-REP/CoppeliaSim, MuJoCo, and PyBullet. These sources provide a research-facing view of the domain.

Academic sources are valuable because they connect software packages to published research contexts. However, they are not sufficient by themselves. Papers may emphasize packages relevant to a specific research question, robot type, or time period. Some active tools may not appear in the selected papers, while older tools may remain visible because they were important at the time of publication. For this reason, academic sources were combined with other discovery sources.

\subsection{Community-Curated Sources}
\label{subsec:community-curated-sources}

Community-curated sources were used to capture a broader view of robotics software practice. The source catalogue includes GitHub topic pages, "awesome" lists, robotics software lists, best-of lists, and other community-maintained aggregations. Examples include sources focused on motion planning, robotics simulation, robotics libraries, ROS 2, Webots, multibody dynamics simulation, and robotic tooling.

These sources are useful because they aggregate software that practitioners and community contributors consider relevant. They can reveal projects that may not be emphasized in academic surveys, and they can expose relationships between simulators, planning libraries, middleware, physics engines, and tools. At the same time, community-curated sources vary in maintenance quality, inclusion criteria, and update frequency. They were therefore treated as discovery aids rather than definitive evidence of software quality.

\subsection{LLM-Assisted Sources}
\label{subsec:llm-assisted-sources}

LLM-generated software lists were used as an auxiliary source for candidate discovery. This use of LLMs is a modification of the original State-of-the-Practice methodology, and it should be described transparently. The LLM-generated lists were not treated as authoritative and did not determine the final software set by themselves. Instead, they were used alongside academic and community-curated sources to broaden the search for candidate software.

The rationale for including LLM-assisted discovery is that large language models can surface software names that appear across public information. However, LLM output can be incomplete, outdated, or inaccurate. For this reason, software names from LLM-assisted sources require consolidation, filtering, and verification against public evidence before they can support thesis claims. In this project, LLM-assisted sources contribute to candidate discovery but not to final measurement claims.

\section{Mention Count Ranking}
\label{sec:mention-count-ranking}

After collecting candidate software names from the source categories, the project consolidated equivalent names and counted how often each software package appeared across the collected sources. The consolidated ranking is recorded in \path{examples/software-list-counting-result.xlsx}, in the worksheet \texttt{final.m}. The worksheet contains the columns \texttt{Rank}, \texttt{Software/Framework/Engine}, and \texttt{Mention Count}.

The top entries in the consolidated ranking include Gazebo with 22 mentions, Webots with 16 mentions, Bullet/PyBullet with 16 mentions, CoppeliaSim/V-REP with 13 mentions, MuJoCo with 12 mentions, ODE with 10 mentions, ROS/ROS 2 with 10 mentions, and OpenRAVE with 9 mentions. These counts help organize the candidate list and make the selection process more transparent.

Mention count is only a rough source-appearance signal. It should not be interpreted as a direct measure of software quality, user adoption, research impact, maintainability, or suitability for a particular robotics task. A package can appear frequently because it is historically important, widely discussed, broadly useful, or simply visible in the selected sources. Conversely, a specialized or newer package may be underrepresented. The ranking is therefore an input to candidate filtering, not a final evaluation.

\section{Filtering Criteria}
\label{sec:filtering-criteria}

The ranked candidate list was filtered to identify software packages suitable for repository-based State-of-the-Practice measurement. In the Domain X methodology, the initial filters are scope, usage, and age. In this project those filters are implemented through the thesis-specific constraints of domain fit, open-source availability, repository-based measurability, and activity or maintenance status. Copies of the initial and filtered lists should be preserved for traceability.

\subsection{Open Source Availability}
\label{subsec:open-source-availability}

Open-source availability was required because the study depends on public software artifacts. The Domain X candidate criteria require viewable source code and prefer software for which repository-based empirical measures are available. Source code, repository history, issue trackers, documentation, and release information provide evidence for the measurements used later in the report. If a software package has important proprietary core components, it cannot be measured consistently using the same repository-mining approach as fully open-source packages.

This criterion led to the exclusion of several important robotics tools. The project materials identify MATLAB/Simulink, Unity, NVIDIA Isaac Sim, and RaiSim as important in the broader robotics ecosystem but unsuitable for consistent repository-based measurement because their core components are not open source in the required way. These exclusions should not be read as claims that the tools are unimportant. They reflect the methodological constraints of this study.

\subsection{Repository-Based Measurability}
\label{subsec:repository-based-measurability}

Repository-based measurability requires that enough public evidence be available to apply the measurement process. Relevant evidence may include source repositories, commit history, issue data, documentation, release information, build or installation instructions, test artifacts, and project metadata. Packages without sufficient public evidence cannot be measured consistently. This criterion corresponds to the Domain X preference for GitHub-style repositories or equivalent sources from which empirical measures can be collected.

This criterion also affects how multi-repository projects are handled. Some robotics software systems are distributed across multiple repositories or organizations. In such cases, the study must define which repositories are measured and preserve that decision as part of the measurement record. The same principle applies to projects that have changed names, migrated repositories, or split into successor projects.

\subsection{Activity and Maintenance Status}
\label{subsec:activity-maintenance-status}

Activity and maintenance status were considered because the project focuses on contemporary software practice. In the Domain X methodology, age is one of the initial filtering criteria, measured by the last date when a change was made, while allowing exceptions for older packages that are still highly recommended or currently used. Tools that are no longer maintained may still be historically important, but they may not provide useful evidence about current practice. The project materials identify MRDS and USARSim as examples of packages excluded because they are no longer actively maintained.

Activity-related evidence should be treated as a snapshot. Repository activity, last commit date, issue counts, and release status can change after measurement. The report should therefore record the timing and source of such measurements when the measurement data is finalized.

\section{Final Software Set}
\label{sec:final-software-set}

After ranking and filtering, the working candidate software list includes Gazebo, Webots, Bullet/PyBullet, CoppeliaSim/V-REP, MuJoCo, ODE, ROS 2, OpenRAVE, PhysX open-source SDK, AirSim, Drake, DART, CARLA, MORSE, Simbody, OMPL, Pinocchio, OROCOS, SOFA, MoveIt!, GraspIt!, Habitat, Project Chrono, Flightmare, MRPT, Newton Dynamics, YARP, and OpenRTM-aist.

This set contains software with different roles. Most entries are associated with simulation, physics, dynamics, planning, or robotics infrastructure, but they are not all direct substitutes. ROS 2, MoveIt!, and OMPL are especially important boundary cases because they are not full simulators in the same sense as tools such as Gazebo or Webots. They are retained because the study scope is robotics software for motion planning and simulation, not only simulator applications.

Several excluded packages should be discussed explicitly in the final thesis. MATLAB/Simulink was excluded because it is not open source and does not support the same public repository-based measurement process. Unity, NVIDIA Isaac Sim, and RaiSim were excluded because their core components are not open source enough for consistent repository mining. MRDS and USARSim were excluded because they are no longer actively maintained. GPMP2 and CHOMP were discussed but not added explicitly because doing so would deviate from the established candidate selection process.

\section{Measurement Procedure}
\label{sec:measurement-procedure}

The measurement procedure follows the Domain X State-of-the-Practice method for collecting structured evidence about software packages in a selected domain. For each selected package, the study should gather source code, documentation, publications when applicable, repository metadata, issue-tracker evidence, release information, and installation evidence. The measurement record should also identify the specific repository or repositories measured, since some robotics projects are distributed across multiple repositories.

The main instrument is the Domain X Appendix B measurement template. The template contains 108 questions grouped under summary information, repository metrics, and nine quality categories: installability; correctness and verifiability; surface reliability; surface robustness; surface usability; maintainability; reusability; surface understandability; and visibility/transparency. Each package should be measured using the same template so that later comparison does not depend on ad hoc observations.

Repository metrics should be collected separately from the quality impressions. GitHub-style metrics include stars, forks, watchers, open and closed pull requests, issue counts, developers, releases, and last commit dates. GitStats-style metrics include binary files, added and deleted lines, and commit activity over time. The scc-style metrics include text-based files, total lines, code lines, comment lines, and blank lines. Processed measures include whether the project appears alive or dead under the 18-month activity rule, the percentage of identified issues closed, and the percentage of code that is comments.

The installation-based parts of the template should be performed in a controlled environment and recorded as a snapshot. The Domain X method expects installation on a virtual machine and recommends limiting installation effort so that the method remains feasible across many packages. Installation observations should record the operating system, dependency instructions, installation steps, validation procedure, and whether errors are descriptive and recoverable.

Several measurement questions require careful interpretation. The last commit date should be treated as a snapshot at the time of measurement. The percentage of identified issues that are closed should be computed consistently, such as closed issues divided by total identified issues when GitHub issue data is used. Publication counts may require a defined scoping rule, especially for widely used software. Evidence that performance was considered should refer to explicit evidence related to speed, storage, throughput, benchmarks, papers, documentation, or source code, rather than unsupported assumptions. Because performance evidence is a summary-information field in Appendix B, it should not be presented as one of the nine top-level quality categories.

After the quality measures are collected, the Domain X methodology uses the measurement-template impression calculator and AHP to support ranking across the nine quality categories. In this thesis, any AHP ranking should be reported only after the underlying measurements are available and checked. The ranking should be interpreted as an analysis aid for comparing evidence under the chosen criteria, not as a universal claim that one robotics package is best for every use case.

\section{Commonality Analysis Procedure}
\label{sec:commonality-analysis-procedure}

The commonality analysis identifies shared and variable capabilities across the selected software packages. It is intended to explain the domain and supplement repository-based metrics. The analysis should be based on checked evidence from documentation, repositories, papers, or other reliable project materials.

Following the Domain X methodology, the analysis should treat the selected packages as members of a software family and document four kinds of information. First, it should list commonalities: goals, assumptions, functions, artifacts, or domain concepts that appear across multiple family members. Second, it should list variabilities: places where family members differ. Third, it should describe parameters of variation, meaning the possible values for each variability. Fourth, where appropriate, it should record binding time, such as whether a variation is fixed at specification time, build time, configuration time, or run time.

The procedure should first group the selected packages into broad software roles, such as robotics simulators, physics and dynamics libraries, and planning or middleware tools. It should then identify capabilities that appear across multiple tools, such as support for robot models, simulated environments, motion planning workflows, dynamics or physics representation, APIs, documentation, and integration with robotics ecosystems. Finally, it should record variations, such as differences in scope, supported platforms, licensing, interfaces, and intended use cases.

Because the selected software set is heterogeneous, the commonality analysis should avoid forcing all tools into a single comparison template. The purpose is to describe shared concerns and meaningful differences, not to claim that every selected package provides the same type of functionality.

\section{Interview Component}
\label{sec:interview-component}

The Domain X methodology includes a developer survey or interview component with semi-structured questions about developer background, users, project organization, software qualities, pain points, documentation, maintainability, understandability, usability, and reproducibility. The project materials indicate that interviews with software developers, maintainers, or programmers may be part of the broader State-of-the-Practice exercise. The interview component is associated with research ethics approval through MacREM. At the time of this draft, the available materials do not provide interview transcripts, participant responses, approved summaries, or final interview findings.

For this reason, this report treats the interview component as planned or pending. Later revisions may describe interview recruitment, consent, protocols, confidentiality, and qualitative analysis if approved materials become available. No interview findings should be reported unless they are supported by actual approved interview data.
