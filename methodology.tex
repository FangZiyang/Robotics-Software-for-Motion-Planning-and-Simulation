\section{Domain Selection}
\label{sec:domain-selection}

The selected domain is robotics software for motion planning and simulation. The scope covers software used in robotics workflows, including simulation, planning, dynamics, middleware, and related support tools. Earlier project discussions considered inverse kinematics software, but later communication established the broader domain used in this thesis.

The boundary is broader than "robotics simulators" alone. Many robotics workflows combine a simulator with planning libraries, physics engines, middleware, and supporting tools. Restricting the study to full simulators would make comparison simpler, but it would omit software central to motion planning and simulation practice. The broader scope better matches the project goal as long as the thesis distinguishes software roles.

Domain selection also follows the Domain X methodology. The methodology expects a domain with a stable technical basis, an identifiable community, open-source software options, and enough candidate packages to support comparison. Motion planning and simulation are established robotics topics, and the project materials identify a substantial set of candidate packages with public artifacts.

The methodology also expects a domain expert to vet the software list, domain analysis, and later ranking. The project materials identify Dr. Matthew Giamou as the domain expert involved in scope and candidate-list discussions. Material awaiting further expert review is marked as pending rather than final.

\section{Candidate Software Discovery}
\label{sec:candidate-software-discovery}

Candidate discovery used three source categories: academic sources, community-curated sources, and LLM-assisted sources. Using more than one category reduced dependence on any single search method. The resulting list was consolidated and filtered before measurement.

\subsection{Academic Sources}
\label{subsec:academic-sources}

Academic sources identified software packages that appear in research discussions of robotics simulators, simulation tools, and related challenges. The source catalogue includes survey-style or comparative papers that mention packages such as Gazebo, Webots, V-REP/CoppeliaSim, MuJoCo, and PyBullet. These sources give a research-facing view of the domain.

Academic sources connect software packages to published research contexts, but they are not sufficient by themselves. Papers may emphasize packages relevant to a specific research question, robot type, or time period. Some active tools may not appear in the selected papers, while older tools may remain visible because they mattered when the paper was written. Academic sources were combined with other discovery sources.

\subsection{Community-Curated Sources}
\label{subsec:community-curated-sources}

Community-curated sources broadened the view of robotics software practice. The source catalogue includes GitHub topic pages, "awesome" lists, robotics software lists, best-of lists, and other community-maintained aggregations. Examples include sources focused on motion planning, robotics simulation, robotics libraries, ROS 2, Webots, multibody dynamics simulation, and robotic tooling.

These sources collect software that practitioners and community contributors consider relevant. They can reveal projects not emphasized in academic surveys and show relationships between simulators, planning libraries, middleware, physics engines, and tools. They also vary in maintenance quality, inclusion criteria, and update frequency. For that reason, they were used as discovery aids rather than evidence of software quality.

\subsection{LLM-Assisted Sources}
\label{subsec:llm-assisted-sources}

LLM-generated software lists were used as an auxiliary source for candidate discovery. This modifies the original State-of-the-Practice methodology, so the thesis records it explicitly. The LLM-generated lists were not authoritative and did not determine the final software set. They broadened the search alongside academic and community-curated sources.

The reason for including LLM-assisted discovery is practical: large language models can surface software names that appear across public information. Their output can also be incomplete, outdated, or wrong. Names from these sources therefore require consolidation, filtering, and verification against public evidence before they can support thesis claims. In this project, LLM-assisted sources contribute to candidate discovery, not to final measurement claims.

\section{Mention Count Ranking}
\label{sec:mention-count-ranking}

After collecting candidate names, the project consolidated equivalent names and counted how often each package appeared across the sources. The consolidated ranking is recorded in \path{examples/software-list-counting-result.xlsx}, in the worksheet \texttt{final.m}. The worksheet contains the columns \texttt{Rank}, \texttt{Software/Framework/Engine}, and \texttt{Mention Count}.

The top entries in the consolidated ranking include Gazebo with 22 mentions, Webots with 16 mentions, Bullet/PyBullet with 16 mentions, CoppeliaSim/V-REP with 13 mentions, MuJoCo with 12 mentions, ODE with 10 mentions, ROS/ROS 2 with 10 mentions, and OpenRAVE with 9 mentions. These counts help organize the candidate list and make the selection process more transparent.

Mention count is a source-appearance signal. It is not a measure of software quality, user adoption, research impact, maintainability, or suitability for a robotics task. A package can appear frequently because it is historically established, widely discussed, broadly useful, or simply visible in the selected sources. A specialized or newer package may be underrepresented. The ranking is an input to candidate filtering, not a final evaluation.

\section{Filtering Criteria}
\label{sec:filtering-criteria}

The ranked candidate list was filtered to identify packages suitable for repository-based State-of-the-Practice measurement. In the Domain X methodology, the initial filters are scope, usage, and age. In this project, those filters are implemented through domain fit, open-source availability, repository-based measurability, and activity or maintenance status. The initial and filtered lists are preserved for traceability.

\subsection{Open Source Availability}
\label{subsec:open-source-availability}

Open-source availability was required because the study depends on public software artifacts. The Domain X candidate criteria require viewable source code and prefer software with repository-based empirical measures. Source code, repository history, issue trackers, documentation, and release information provide evidence for later measurements. A package with proprietary core components cannot be measured consistently with the same repository-mining procedure used for open-source packages.

This criterion led to the exclusion of several prominent robotics tools. The project materials identify MATLAB/Simulink, Unity, NVIDIA Isaac Sim, and RaiSim as prominent in the robotics ecosystem but unsuitable for consistent repository-based measurement because their core components are not open source in the required way. These exclusions reflect method constraints, not the value of the tools.

\subsection{Repository-Based Measurability}
\label{subsec:repository-based-measurability}

Repository-based measurability requires enough public evidence to apply the measurement process. Relevant evidence may include source repositories, commit history, issue data, documentation, release information, build or installation instructions, test artifacts, and project metadata. Packages without sufficient public evidence cannot be measured consistently. This criterion corresponds to the Domain X preference for GitHub-style repositories or equivalent sources from which empirical measures can be collected.

This criterion also affects how multi-repository projects are handled. Some robotics software systems are distributed across multiple repositories or organizations. In such cases, the study must define which repositories are measured and preserve that decision as part of the measurement record. The same principle applies to projects that have changed names, migrated repositories, or split into successor projects.

\subsection{Activity and Maintenance Status}
\label{subsec:activity-maintenance-status}

Activity and maintenance status were considered because the project focuses on contemporary software practice. In the Domain X methodology, age is one of the initial filtering criteria, measured by the last date when a change was made, with exceptions for older packages that are still recommended or currently used. Tools that are no longer maintained may have historical value, but they provide weaker evidence about current practice. The project materials identify MRDS and USARSim as packages excluded for this reason.

Activity-related evidence is a snapshot. Repository activity, last commit date, issue counts, and release status can change after measurement. The finalized measurement data should record the timing and source of these observations.

\label{sec:final-software-set}

After ranking and filtering, the working candidate software list includes Gazebo, Webots, Bullet/PyBullet, CoppeliaSim/V-REP, MuJoCo, ODE, ROS 2, OpenRAVE, PhysX open-source SDK, AirSim, Drake, DART, CARLA, MORSE, Simbody, OMPL, Pinocchio, OROCOS, SOFA, MoveIt!, GraspIt!, Habitat, Project Chrono, Flightmare, MRPT, Newton Dynamics, YARP, and OpenRTM-aist.

This set contains software with different roles. Most entries are associated with simulation, physics, dynamics, planning, or robotics infrastructure, but they are not direct substitutes. ROS 2, MoveIt!, and OMPL are boundary cases because they are not full simulators in the same sense as Gazebo or Webots. They are retained because the study scope is robotics software for motion planning and simulation, not simulator applications alone.

The final thesis needs to discuss several excluded packages explicitly. MATLAB/Simulink was excluded because it is not open source and does not support the same public repository-based measurement process. Unity, NVIDIA Isaac Sim, and RaiSim were excluded because their core components are not open source enough for consistent repository mining. MRDS and USARSim were excluded because they are no longer actively maintained. GPMP2 and CHOMP were discussed but not added explicitly because doing so would deviate from the established candidate selection process.

\section{Measurement Procedure}
\label{sec:measurement-procedure}

The measurement procedure follows the Domain X method for collecting structured evidence about software packages in a selected domain. For each selected package, the study gathers source code, documentation, publications when applicable, repository metadata, issue-tracker evidence, release information, and installation evidence. The measurement record also identifies the repository or repositories measured, since some robotics projects are distributed across multiple repositories.

The main instrument is the Domain X Appendix B measurement template. The template contains 108 questions grouped under summary information, repository metrics, and nine quality categories: installability; correctness and verifiability; surface reliability; surface robustness; surface usability; maintainability; reusability; surface understandability; and visibility/transparency. Applying the same template to each package keeps later comparison from becoming a set of ad hoc observations.

Repository metrics are collected separately from the quality impressions. GitHub-style metrics include stars, forks, watchers, open and closed pull requests, issue counts, developers, releases, and last commit dates. GitStats-style metrics include binary files, added and deleted lines, and commit activity over time. The scc-style metrics include text-based files, total lines, code lines, comment lines, and blank lines. Processed measures include whether the project appears alive or dead under the 18-month activity rule, the percentage of identified issues closed, and the percentage of code that is comments.

The installation-based parts of the template are performed in a controlled environment and recorded as a snapshot. The Domain X method expects installation on a virtual machine and limits installation effort so the method remains feasible across many packages. Installation observations record the operating system, dependency instructions, installation steps, validation procedure, and whether errors are descriptive and recoverable.

Several measurement questions require interpretation rules. The last commit date is a snapshot at the time of measurement. The percentage of identified issues closed is computed consistently, such as closed issues divided by total identified issues when GitHub issue data is used. Publication counts may require a defined scoping rule, especially for widely used software. Evidence that performance was considered comes from explicit references to speed, storage, throughput, benchmarks, papers, documentation, or source code rather than from assumptions. Because performance evidence is a summary-information field in Appendix B, it is not reported as one of the nine top-level quality categories.

After the quality measures are collected, the Domain X methodology uses the measurement-template impression calculator and AHP to support ranking across the nine quality categories. In this thesis, any AHP ranking appears only after the underlying measurements are available and checked. The ranking is an analysis aid for comparing evidence under the chosen criteria, not a universal claim that one robotics package is best for every use case.

\section{Commonality Analysis Procedure}
\label{sec:commonality-analysis-procedure}

The commonality analysis identifies shared and variable capabilities across the selected packages. It explains the domain and supplements repository-based metrics. The analysis is based on checked evidence from documentation, repositories, papers, or other reliable project materials.

Following the Domain X methodology, the analysis should treat the selected packages as members of a software family and document four kinds of information. First, it should list commonalities: goals, assumptions, functions, artifacts, or domain concepts that appear across multiple family members. Second, it should list variabilities: places where family members differ. Third, it should describe parameters of variation, meaning the possible values for each variability. Fourth, where appropriate, it should record binding time, such as whether a variation is fixed at specification time, build time, configuration time, or run time.

The procedure first groups the selected packages into broad software roles, such as robotics simulators, physics and dynamics libraries, and planning or middleware tools. It then identifies capabilities that appear across multiple tools, such as support for robot models, simulated environments, motion planning workflows, dynamics or physics representation, APIs, documentation, and integration with robotics ecosystems. It also records variations in scope, supported platforms, licensing, interfaces, and intended use cases.

Because the selected set is heterogeneous, the commonality analysis does not force all tools into a single comparison template. Its purpose is to describe shared concerns and meaningful differences, not to claim that every package provides the same functionality.
